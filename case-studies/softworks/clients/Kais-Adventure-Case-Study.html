  "id": "B9GGWa8swkrk",           // Unique token

Device Fingerprinting (Soft Lock)

To prevent guests from sharing links widely, we use "soft" device fingerprinting:

  1. First use: We record a fingerprint of the device (browser + screen + timezone)
  2. Subsequent uses: If a different device tries to use the link, it shows "locked"
  3. Admin override: Hui can unlock any invite from the admin panel
ğŸ” Why "Soft" Lock?

We don't want to frustrate guests who switch phones or browsers. The lock is just a gentle deterrent against sharing. If someone really wants to share their link with a friend, Hui can add more credits instead of fighting it.

ğŸ”§ Challenges & Solutions

Every project hits unexpected problems. Here are the biggest challenges we faced and how we solved them.

Challenge 1: Gemini Quota Limits

âŒ The Problem

Gemini's free tier only allows 30 image generations per day. A party with 30 guests, each trying 2-3 themes, would hit the limit in the first hour.

âœ… The Solution

Added Replicate FLUX as automatic fallback. When Gemini fails, the system seamlessly switches to FLUXâ€”guests never notice. FLUX costs ~$0.04/generation but has no daily limit.

Challenge 2: Mobile Safari Scroll Issues

âŒ The Problem

The consent modal used nested scrollable containers. On iOS Safari, users couldn't scroll to read the full terms. The checkbox was also too small to tap reliably.

âœ… The Solution

Rewrote the modal as a full-page layout with sticky header/footer. Made the entire checkbox row tappable (not just the tiny box). Tested on real iPhones before deploying.

Challenge 3: iOS Image Saving

âŒ The Problem

The standard web "download" approach doesn't work on iOS Safari. Clicking a download link would open a new tab with the image, then... nothing. Users couldn't save it.

âœ… The Solution

Created a two-step approach: First, try the Web Share API (opens iOS share sheet). If that fails, show a full-screen modal with the image and clear instructions: "Long press the image, then tap 'Add to Photos'."

Challenge 4: Face Preservation Quality

âŒ The Problem

Early tests showed the AI sometimes "smoothing out" children's faces or making them look older. Parents want their kid to look like their kid.

âœ… The Solution

Added explicit instructions in every prompt: "CRITICAL: Preserve the exact face, skin tone, and expression. Maintain facial features precisely as shown." This dramatically improved face accuracy.

Challenge 5: Admin UX for Non-Technical User

âŒ The Problem

Initial admin panel had too many buttons. The "+5 credits" button gave no feedback. WhatsApp sharing required multiple taps.

âœ… The Solution

Simplified to three actions per guest: Copy, +10, Edit. Added toast notifications that confirm actions AND offer to send the link immediately. One tap to copy, two taps to WhatsApp.

Every project teaches something. Here are the key takeaways from building Kai's Adventure.

Technical Lessons

ğŸ“±
Test on Real Devices

Simulators lie. The scroll issues and save problems only appeared on actual iPhones. Always test on the devices your users will actually use.

ğŸ”„
Build Fallbacks from Day One

We added FLUX fallback on Day 2 after hitting Gemini's quota. Should have designed for multiple providers from the start. AI APIs are unpredictable.

ğŸ’¾
Mobile Save is Hard

There's no universal "download file" on mobile browsers. Plan for Share API + long-press fallback from the beginning.

ğŸ“
Prompts Are Everything

The difference between "bad AI output" and "magical result" is prompt engineering. Spend time on detailed, explicit instructions.

Product Lessons
